# 《滴流 Drop Flow》作品全脉络与版本归档

## 分享附录 A：如何用 30-60 分钟讲清 SpacePort 项目脉络

### 1. 一句话主线

这场分享不只是讲《滴流》这一个作品，而是借《滴流》去讲 VIRTURA / SpacePort 这条线如何一步步收束：从更早期围绕“虚拟航线 / VRChat / 太空站想象”的空间叙事出发，到 GitHub 上尝试同时承载作品、知识、理论、组织与世界观，再到现在明确把 `VIRTURA-SpacePort` 收回为“公开入口、档案数据库、叙事门面”，把真正长期推进的工具主线收敛到 `SceneForge`。

### 2. 适合对外讲的四段式结构

#### 第一段：前史与原始冲动（建议 5-10 分钟）

- **关键词**：虚拟航线、VRChat、太空站、可体验数字空间
- **核心讲法**：最开始吸引我们的，不是“做一个软件产品”，而是“能不能把数字空间当真实空间来对待”。那时候更像是在寻找一种新的空间经验和叙事容器，所以会自然生长出虚拟航线、太空站、沉浸式体验、世界观包装等很多分支。
- **这一阶段的价值**：它建立了我们最早的世界观直觉，也让我们明确了自己真正关心的是“空间感知”“数字自然”“沉浸式体验”。
- **这一阶段的问题**：概念很多、容器很多、方向很多，但真正能稳定推进和反复验证的工具不够清晰，所以会出现一种“发散有余、收束不足”的状态。

#### 第二段：GitHub 化与“智力装备”阶段（建议 8-15 分钟）

- **关键词**：GitHub、公开档案、知识网络、理论整理、智力装备
- **核心讲法**：进入 GitHub 阶段后，我们开始把之前分散的实践和想法转成可以公开积累的结构，包括作品归档、知识网络、理论文稿、团队入口、项目导航。这一步的进展是：很多东西第一次被写清楚、归档下来、彼此建立连接，不再只存在于脑中的构想。
- **可以这样总结**：如果说早期是在“造世界”，那么这一阶段是在“给世界建立文档、地图和认知装备”。
- **这一阶段的问题**：虽然比之前更系统，但 `SpacePort` 一度仍然像“主港 / 容器 / 大系统”，承载了太多不同层级的东西。它可以解释很多，但不够利于执行，也容易把外部理解引向一个过大的平台叙事。

#### 第三段：从平台容器回到清晰分工（建议 8-15 分钟）

- **当前清晰口径**：
  - `VIRTURA-SpacePort`：团队公开入口、档案数据库、叙事门面
  - `SceneForge`：唯一核心产品主线
  - `RepoForge` / `Forge`：治理与工作流层
- **核心变化**：不再试图让一个仓库同时承担“组织入口 + 理论容器 + 产品主仓 + 世界观主港”的全部角色，而是承认不同仓库有不同职责。
- **你可以直接讲的一句话**：我们不是把项目做大了，而是终于把边界做清楚了。
- **最重要的转变**：从“很多概念性的可能性”切回“一个真正要做成的工具”，目前这个工具就是 `SceneForge`。

#### 第四段：为什么《滴流》是理解这条线的关键作品（建议 10-20 分钟）

- 《滴流》不是孤立作品，而是这条线最能说明问题的样本。
- 它能把几个层面同时串起来：
  - **空间观念**：从“进入一个数字空间”到“无边际媒体内容”
  - **技术演进**：从实时音画实验到多场景适配
  - **叙事演进**：从持续流动的感觉，到“数字焦虑→数字共生”
  - **工具需求**：每一次现场落地都在反过来逼迫我们思考，真正需要的不是更多概念，而是更稳定、更可复用的工具系统
- 所以，《滴流》既是作品档案，也是一条产品线为什么必须收束为工具的证据。

### 3. 30 分钟版 / 60 分钟版建议

| 时长 | 讲法重点 | 建议分配 |
|------|---------|---------|
| 30 分钟版 | 以“项目为何收束”为主，作品为例证 | 前史 5 分钟 + GitHub 化 8 分钟 + 当前架构 7 分钟 + 《滴流》案例 10 分钟 |
| 45 分钟版 | 项目脉络与《滴流》并重 | 前史 8 分钟 + GitHub 化 10 分钟 + 当前架构 10 分钟 + 《滴流》案例 12 分钟 + Q&A 5 分钟 |
| 60 分钟版 | 加入版本细节、方法论和工具判断 | 前史 10 分钟 + GitHub 化 / 智力装备 15 分钟 + 当前架构与仓库分工 10 分钟 + 《滴流》版本演进 20 分钟 + Q&A 5 分钟 |

### 4. 可以直接拿去讲的结尾

> 如果用一句话总结现在的 SpacePort 项目脉络，那就是：我们经历过一次非常典型的“先发散、再收束”的过程。早期我们更像在为未来数字空间寻找语言，后来我们尝试用 GitHub 去承载作品、理论、组织与世界观，而现在我们终于更明确地知道，入口归入口，档案归档案，真正需要持续打磨成工具的主线，只保留 `SceneForge`。

## 分享附录 C：理论收束、知识网络与组织宣传口径

### 1. 为什么这一段必须把理论讲进去

如果只讲作品和仓库结构，`SpacePort` 容易被理解成“一个做过很多东西、现在决定收缩的 GitHub 项目”。但实际上，这条线的真实价值不只是做了多少作品，而是这些作品、练习、工作坊、理论文稿，正在慢慢被收束成一套更有效的创作与组织系统。

可以把当前的理论支撑简化成三层：

- **空间原生创作框架**：回答“视觉创作者如何从认知走到空间作品落地”，强调创作优先、最小闭环、工程化拆解。
- **工程控制论**：回答“如何把一次次实践经验变成可反馈、可调参、可迭代的控制系统”，避免一直停留在概念描述层。
- **知识网络 / 智力装备逻辑**：回答“为什么短期工作坊、项目复盘、工具经验不能只写成文章，而要变成可被再次调用的应用态知识”。

换句话说，理论在这里不是装饰，不是为了显得宏大，而是为了把原本零散的感知经验、技术试验、教学经验与组织经验，压缩成一个能持续生长的框架。

### 2. “感知体验设计”怎么从概念变成有效控制系统

你可以直接这样讲：我们早期对“感知体验设计”的理解，确实更偏概念，知道自己想要什么氛围、什么空间感、什么人机关系，但不够知道如何稳定地复现它、拆解它、教学它。现在这一点正在改变。

所谓“变成有效的控制系统”，就是把感知体验拆成几个可以持续调试的层：

- **目标层**：想让观众进入什么状态，比如安静、下坠、被包裹、被唤醒、从焦虑走向共生。
- **变量层**：真正参与塑造体验的变量，包括光影、景深、节奏、声场、空间边界、交互触发、设备形态、叙事顺序。
- **执行层**：通过 demo、排练、现场演出、装置适配、工作坊练习，把这些变量做成可以重复调用的动作和模块。
- **反馈层**：从现场事故、观众反应、排练笔记、工作坊卡点、设备限制里回收信息，再反向修改系统。

这样一来，“感知体验设计”就不再只是抽象描述，而变成一个能不断被验证的闭环。

### 3. 《滴流》正好是这个控制系统最清楚的案例

- **1.0 原型版**：解决的是“实时音画能不能成立”，是底层技术 demo，也是对“drop之后的状态”这一创作哲学的第一次确定。
- **2.0 UFO 内部版**：开始测试多素材协同、AI内容、现实扫描和现场执行问题，让系统第一次暴露出标准化不足。
- **3.0 Rooooooom719 版**：开始把“感知体验”做成更成熟的公开演出语言，验证它是否能在真实观众前成立。
- **杭州双年展版**：第一次把“从屏幕框定时代到无边际媒体时代”讲成完整的人机关系叙事。
- **重庆版**：进一步验证同一套内容结构能否跨空间装置迁移。

所以《滴流》不是一串孤立版本，而是一条非常典型的“从技术 demo 到创作哲学，再到空间叙事与适配方法”的收束路径。

### 4. 知识网络为什么一定要补上“7天工作坊之后”的部分

7天工作坊的文档和批评其实很关键，因为它直接暴露了一个现实问题：**短期工作坊可以制造高强度进入感，但如果没有后续支撑，学员很容易学完即忘，项目经验也很难沉淀成长期能力。**

当前这套理论收束的意义，就在于把这个缺口补上：

- 工作坊负责**激活**：让人快速进入一个方向，尽快看到成果。
- 知识网络负责**存储**：把知识点、案例、问题、模板结构化保存下来。
- 智力装备负责**应用**：把知识网络按具体困境重新组装成“当前就能用”的急救包、路径包、模板包。
- 工程控制论负责**反馈迭代**：保证整个系统不是一次性发放，而是能根据真实使用情况持续调参。

也就是说，7天工作坊不是终点，而只是这套系统里的“第一次点火”。

### 5. 之前的 VIRTURA 太空站内容，现在可以怎么重新解释

这一段也很重要，因为它能帮你把“早期发散”讲得更成立。

- 早期的太空站、虚拟航线、可体验数字空间，并不只是无效膨胀，它们是你们在为未来数字空间寻找容器、语言和场景原型。
- 问题不在于这些探索本身没价值，而在于当时还没有足够清晰的工具主线和反馈机制去承接它们。
- 现在回头看，这些内容可以被重新理解为：**VIRTURA 在不同阶段做过的一系列感知实验场、叙事实验场、空间原型场。**
- 今天真正成熟的做法，是把这些前史拆开处理：
  - 作品与前史，放进 `SpacePort` 的公开档案层
  - 理论与方法，放进知识网络与我们的理论
  - 真正持续推进的产品能力，只收束到 `SceneForge`

这样讲，早期内容就不再像“做散了”，而更像“前期采样很多，现在把这些采样压成了一套更清楚的结构”。

### 6. 可直接用于组织宣传工作讲稿的版本

> 如果只看表面，VIRTURA / SpacePort 像是一个从很多概念慢慢收缩回一个工具主线的 GitHub 项目；但如果往里看，它其实经历的是一个更有代表性的过程：我们先在虚拟航线、太空站、可体验数字空间这些想象里寻找未来空间媒介的语言，再通过 GitHub 把作品、知识网络、理论和组织经验逐步归档下来，最后才越来越明确，哪些内容应该作为入口和档案保留，哪些内容应该被收束成真正可持续推进的工具。  
>   
> 在这个过程中，《滴流》是最关键的一条样本线，因为它完整呈现了我们如何从技术 demo 走到创作哲学，如何把“持续流动”“无边际媒体”“数字焦虑到数字共生”这些感知命题，一步步转成可验证的演出结构、可迁移的空间内容和可复盘的方法论。  
>   
> 而知识网络、7天工作坊反思、工程控制论这些理论层，并不是悬空概念，它们的作用正是把原本零散的创作经验、教学经验、现场反馈和工具判断，收束成一套有效的控制系统，让 VIRTURA 不只是有作品、有想法，也逐步拥有了可以继续生长的组织方法。

### 7. 如果你想把这段讲得更短

可以压缩成一句：

> 我们现在不是在继续扩张一个“太空港式大系统”，而是在把过去几年关于空间、感知、作品、工作坊和知识网络的探索，收束成一个更有效的组织结构：入口和档案留给 `SpacePort`，理论留在知识网络，真正的工具主线只保留 `SceneForge`。

## 二、各版本详细归档文档

### （一）drop-flow-1.0-prototype.md（滴流 1.0 原型版）

#### 1. 基础信息
- **版本定位**：作品技术原型与视觉底层实验版本，核心验证 "实时音画互动" 技术可行性
- **时间范围**：2025.01-04（核心研发）；2025.04.07（发布技术验证视频）；2025.05.14（发布内部展演实录）
- **核心贡献者**：视觉艺术（钱誉文）、电子音乐（徐昊）

#### 2. 核心技术
- **核心框架**：基于 OSC（Open Sound Control）协议 + Blender Python OSC 插件实现实时控制，结合 Blender 几何节点（模拟节点）搭建生成视觉系统
- **技术亮点**：通过 "blender midi + blender geometry node +（audio2bl）sound bake to curves" 实现音频驱动视觉，超过 300 帧的 mesh 点自动删除仍能稳定 60 帧
- **硬件控制**：引入 Akai APC mini 控制器（推子 / 按钮），实现现场实时操控视觉参数，摆脱纯预渲染视频的限制

#### 3. 核心视觉与叙事
- **视觉核心**：纯黑白点线面阵列，将音乐频谱分析转化为立体、持续运动的圆环阵列，以 "点的轨迹流动" 为核心视觉语言
- **音乐创作**：早期音乐暂命名《由一滴水构成的》，核心表达 "由一个点出发的流动声音"，摒弃传统电子乐 "峰值爆发" 逻辑，聚焦 "持续的流动感"
- **作品命名**：确定 "滴流（Drop Flow）"，核心概念为 "一滴水落下后的平静与持续流动"

#### 4. 关键物料
- 技术验证视频（2025.04.07）：https://www.bilibili.com/video/BV12tRrYTEYB
- 内部展演实录（2025.05.14）：2025 VIRTURA | 滴流 Drop Flow | 上海 UFO Terminal 未知终端 | 沉浸式空间实时视觉演出 | 一种持续性的结构涌现
