>
> 这一版重点补入：GitHub 整体脉络、个人作品年表、团队协作线索、Drop Flow 的正式版本与方法分支、仓库隐藏/归档口径

## 一、先给观众一句总论

如果要用一句话概括我这几年在做什么，我会这样说：

**我不是在同时做一堆分散的项目，而是在持续寻找一种数字空间语言。前几年这条线比较发散，表现为 VIRTURA 太空站、虚拟航线、VRChat、空间叙事、作品实验；现在这条线被逐步收束成三个层面：公开入口与档案层放在 `VIRTURA-SpacePort`，理论和知识网络负责沉淀方法，真正长期推进的工具主线只保留 `SceneForge`。**

《滴流 Drop Flow》是这条线里最关键的样本，因为它不是一个孤立作品，而是一条从技术 demo、生长为创作哲学，再扩展为多场景演出和空间适配方法的母体线。

## 二、开场可以这样讲

大家好，今天这场分享我表面上是在讲《滴流 Drop Flow》，但我真正想讲清楚的是整个 GitHub SpacePort 项目的脉络，以及我这些年个人作品、团队协作和方法论是怎么逐渐收束起来的。

如果只看我 GitHub 上这些仓库，或者只看某一个阶段的作品，很容易觉得它们是分散的：有的是虚拟空间，有的是舞台视觉，有的是沉浸装置，有的是知识文稿，有的是工具仓。

但我现在回头看，会发现它们其实都在围绕同一个问题打转：

**如何把一次性的空间体验、现场演出和感知实验，变成一种可复用、可迁移、可继续生长的数字空间语言。**

所以今天我想讲的不是“我做过哪些项目”这么简单，而是：

- 这条线是怎么从发散走到收束的
- 为什么理论不是空话
- 为什么 GitHub 上会出现这么多仓库
- 为什么现在反而要把很多东西隐藏、冻结、归档
- 为什么最后工具主线只留一个 `SceneForge`
- 为什么《滴流》会成为这条线上最关键的母体作品

## 三、第一层前史：VIRTURA 最早不是工具项目，而是空间语言实验

最开始吸引我的，不是“做软件产品”，而是一个更前置的问题：

**能不能把数字空间当成真实空间来对待。**

所以在最早的 VIRTURA 阶段，项目会自然长出很多今天看上去很大的词：

- 太空站
- 虚拟航线
- VRChat
- 可体验数字空间
- 沉浸式叙事容器
- 空间中的角色和入口感

如果从产品经理视角看，这一阶段确实不够收敛；但如果从创作史角度看，它并不是无效发散，而是在帮我确认几个长期不变的问题：

- 空间感知到底怎么被设计
- 数字自然能不能替代传统场景想象
- 人怎样从看屏幕，转向进入空间
- 影像怎样从画面，变成环境、氛围和关系

所以这一阶段真正留下来的，不是某个“平台”概念，而是一套非常原始但重要的词汇库。

### 这一段可以压缩成一句话

我们一开始不是在做工具，而是在为未来数字空间寻找语言。

## 四、第二层转折：从现场项目里，慢慢长出几条稳定的方法线

如果按作品和实践来看，这条线不是突然在 GitHub 里形成的，而是先在项目里慢慢长出来的。

我会把它大致分成四个阶段。

### 1. 2022：从舞台与屏幕空间开始

这一年更像底座期，主要是在真实场域里处理影像、屏幕、投影和空间关系。

- 2022.09 上海时装周 XTEP-XDNA 22AW
- 2022.10 谢欣舞蹈剧场《四相》《汞》
- 2022.12 深圳光影艺术季 AR 深圳计划

这一阶段最重要的不是风格统一，而是建立了我对“现场成立”的基本判断：

- 影像不是越多越好，而是要在场地里成立
- 画面必须服从空间和观看关系
- 制作不能只停留在审美，还要能完成工程交付

### 2. 2023-2024：从展览与实验过渡到时间、音画、扫描和空间叙事

这个阶段更像过渡带，也是很多后面工作的真正转折点。

可以看到几条线同时出现：

- 《时间操纵者 / Timer》开始把音频、节拍、画面结构组织起来
- `UFO Terminal「加载…」` 的进入，让作品开始进入更实验性的展示语境
- 《Dérive 双城记：Digital Dérive》把扫描、数字空间和空间计算迁移放进一个更明确的作品结构里

对我来说，2024 最关键的变化不是“项目变多了”，而是三个底层方法开始成形：

- 音画联动
- 扫描与记忆空间
- 从屏幕内容转向空间叙事

### 3. 2025：作品密度最高，也是方法真正爆发的一年

这一年里，很多线突然连起来了。

一方面是《滴流》从原型走到公开演出、竞赛、双年展和重庆装置版本；另一方面，其他项目也在继续扩展同一套方法语言：

- 《滴流 Drop Flow》杭州国际电子音乐作曲比赛一等奖
- 首届中国（杭州）艺术与科技国际双年展开幕式《滴流 Drop Flow 2025》
- 重庆 LUMINOUS《滴流》优秀作品奖
- 《未来弦诗》
- 《观察与共生》
- 西安《Digital Garden》
- FutureLab、Vision Pro 迁移等空间计算相关实践
- Kashiwa Daisuke 深圳专场与 Can Festival 舞台视觉制作

这也是为什么我现在更愿意把 2025 看成“方法成形年”，而不只是“项目很多的一年”。

### 4. 2026：开始做真正的结构收束

到了 2026，变化不在于又开了多少新坑，而在于开始明确分层：

- 什么是入口
- 什么是档案
- 什么是理论
- 什么是工具
- 什么该公开
- 什么该冻结
- 什么该私有

这也是 SpacePort 和 GitHub 整体清理的核心背景。

### 这一段可以压缩成一句话

前几年是在项目里长方法，今天是在结构上做收束。

## 五、第三层：为什么 `our-theory` 不是悬空概念

很多人看到知识网络和理论文档，会以为这只是概念整理。但对我来说，这些文档真正的作用，是把之前项目里零散出现的经验压缩成可持续调用的系统。

这里至少有三条关键理论线。

### 1. 空间原生创作框架

这条线回答的是：

**视觉创作者如何从感觉出发，最终做出真正成立的空间作品。**

它不是要制造概念，而是要把创作拆成能执行的小单元：

- 创作优先
- 技术服务表达
- 从最小闭环开始
- 用小范围验证代替空泛设想

### 2. 工程控制论

这条线对我特别重要，因为它解决了“感知体验设计过于概念化”的问题。

过去我们会说氛围、沉浸、数字自然、空间感，但这些词如果不能转化为控制关系，就无法稳定复现，也无法教学，更无法把经验传给团队。

工程控制论提供了一个更有效的看法：

- 目标是什么
- 变量是什么
- 触发方式是什么
- 反馈怎么回收
- 失稳发生在哪
- 怎样调参而不是凭感觉赌运气

这样一来，感知体验设计就不再只是“概念很好听”，而开始变成一种可调、可迭代的控制系统。

### 3. 知识网络 / 智力装备

这条线是对工作坊和短期训练的反思。

我现在越来越清楚，7 天工作坊这类形式只能解决“快速进入”，但解决不了“长期留下来”。

所以一个完整系统至少要有四层：

- 工作坊负责点火
- 知识网络负责存储
- 智力装备负责按问题调用
- 控制系统负责持续反馈和修正

也就是说，知识不该只停留在文章里，而要能变成后来真正有用的装备。

### 这一段可以压缩成一句话

理论的作用，不是让项目显得宏大，而是把经验变成一个能继续使用的系统。

## 六、第四层：为什么 GitHub 上会出现这么多仓库，现在又为什么要隐藏一部分

这一段是今天很重要的一部分，因为它直接关系到外界怎么看 SpacePort。

过去仓库多，不完全是因为方向混乱，而是因为不同阶段承担了不同任务：

- 有些仓库在承担世界观和叙事外壳
- 有些仓库在承担团队入口
- 有些仓库在承担工具实验
- 有些仓库在承担方法论命名
- 有些仓库其实只是某个阶段的临时概念容器

但现在问题也很清楚：

**如果这些仓库全部平铺展示，外界会误以为我在同时运营很多条同等重要的主线。**

这会带来两个后果：

- 叙事显得膨胀
- 真正重要的主线反而被冲淡

所以现在的 GitHub 展示应该被压缩成四层。

### 1. 对外入口层

- `portfolio`：个人单点入口
- `VIRTURA-SpacePort`：团队公开入口、档案数据库、叙事门面

### 2. 核心产品层

- `SceneForge`：唯一核心工具主线

### 3. 治理层

- `RepoForge`：公开治理工具壳
- `Forge`：私有内部治理仓

### 4. 作品与文档层

- 作品型实验仓保留，但不再讲成“大平台”
- 文档和方法仓保留其资料价值，但不再制造重复入口

### 建议弱展示、归档或转私有的仓库

这一部分在直播里可以直接说得很清楚：

- `VIRTURA-Team`：和 `VIRTURA-SpacePort` 的组织入口功能重复，适合迁移后归档或转私有
- `LiveForge`：和 `SceneForge` 的产品叙事重复，适合冻结并入
- `THE-NEXT-ARENA`：平台概念重复，适合归档
- `Virtura-Perceptual-Engine`：更像阶段性命名仓，适合归档或删除
- `Virtura-Perceptual-Toolkit`：如果没有真正独立落地，适合并入文档层或 `SceneForge`

### 我对外会怎么解释这件事

我不会说“以前做错了所以要藏起来”，而会说：

**过去这些仓库各自记录了某个阶段的探索，但现在我们已经更明确地知道，什么是入口，什么是档案，什么是方法，什么是工具。为了让整个 GitHub 对外更清楚，我们会保留有价值的档案，同时把重复入口和重复产品概念收起来。**

### 这一段可以压缩成一句话

仓库不是越多越强，真正成熟的表现，是知道哪些该公开，哪些该收束。

## 八、第六层：团队记录和协作记录怎么讲

讲团队时，不要把它讲成一个固定公司编制，而要讲成一个按项目聚合、按问题组装的创作网络。

### 1. VIRTURA 的特点

VIRTURA 更像一个持续变化的协作结构，而不是一个从头到尾固定不变的团队。

它的价值在于：

- 能围绕作品快速组装
- 能把不同专长的人接到同一个空间问题上
- 能让作品和方法同时推进

### 2. 几个关键协作节点

- UFO Terminal「加载…」：让《滴流》2.0 得到第一次真实实验场
- PRE / Rooooooom719：让《滴流》3.0 形成更成熟的公开演出语言
- 杭州双年展：让《滴流》第一次进入更完整的艺术语境和身体表演结构
- 重庆 LUMINOUS：验证《滴流》在异形屏和不同空间条件下的迁移能力
- Afterlook / 观池空间：把扫描、记忆空间与 workshop 连接起来
- Story App / Vision Pro 相关协作：把内容迁移到空间计算媒介

### 3. 《滴流》中的团队结构非常能说明问题

《滴流》不是一开始就有完整团队，而是随着问题升级，团队也逐步升级：

- 1.0：钱誉文 + 徐昊，解决音画实时是否成立
- 2.0：加入许思慧、黄敏萱，开始处理交互、AI 素材、现场执行
- 3.0：加入马逍遥，公开演出语言更完整
- 双年展版：加入沈轩、王家明、魏琴等，作品进入更明确的展演结构
- 重庆版：6 人核心团队，开始验证跨空间装置的可迁移性

### 这一段可以压缩成一句话

团队不是先搭好再创作，而是在每次更复杂的问题里被重新组织出来。

## 九、第七层：《滴流》为什么是最好的母体案例

《滴流》重要，不只是因为它做了很多版本，而是因为它同时完成了三件事：

- 它验证了实时音画和空间生成
- 它长出了清晰的创作哲学
- 它证明这套语言可以跨场景迁移

### 1. 1.0 原型版：技术与哲学同时诞生

1.0 的核心不是“炫技”，而是非常朴素的问题：

**音乐能不能真的驱动空间中的视觉生成。**

这一版已经确定了《滴流》的核心哲学：

- 不是爆发
- 而是下坠后的平静
- 不是瞬时高潮
- 而是持续流动和持续涌现

同时，几何节点、点云、OSC、音频驱动这些技术语言也开始定型。

### 2. 2.0 UFO 内部实验版：多元素材开始汇流

这一版真正重要的地方，是把几条素材线接在了一起：

- 实时生成
- AI 图像
- 现实扫描
- 现场控制

而且它第一次暴露了系统短板，比如：

- 路径错误
- 长视频卡顿
- 标准化不足

所以它的价值不只是“做出来了”，而是让后面必须开始思考标准化和稳定性。

### 3. 3.0 Rooooooom719 版：公开演出语言成熟

到了 3.0，《滴流》已经不再只是实验。

它开始拥有更明确的演出包装：

- 数字信息流潜艇乘务组
- 360 度包裹式空间感
- 更成熟的角色和观众进入路径

它证明了一件事：前面的技术和叙事，并不是只能留在内部实验里，而是可以对真实观众成立。

### 4. 竞赛版、杭州双年展版、重庆版：从作品走向方法

这一段要重点讲，因为它最能说明《滴流》不是一个单点事件。

竞赛版说明：

- 这套语言可以被压缩成更纯粹的音乐/视觉表达

杭州双年展版说明：

- 这套语言可以进入更完整的人机关系和艺术语境
- 作品不只是流动感，而是在讲数字焦虑、潜入、冲突、共生

重庆版说明：

- 同一套内容结构可以适配异形屏和不同展演空间

也就是说，到这里《滴流》已经不只是作品，而是一种空间内容结构。

## 十、第八层：Drop Flow 有两种“分支”，直播里一定要讲清楚

这一段正好可以回应“为什么它后面会分很多版本、很多合作”的问题。

我建议把《滴流》的分支明确分成两类。

### 第一类：正式版本分支

这是作品本体内部的版本线：

- 1.0 原型版
- 2.0 UFO 内部实验版
- 3.0 Rooooooom719 版
- 竞赛版
- 杭州双年展版
- 重庆展演版
- 2026 实时引擎与空间交互规划版

这条线是在讲作品本体如何升级。

### 第二类：方法外溢分支

这条线更接近你现在想讲的重点。

从创作脉络上看，《滴流》真正留下来的不只是一个作品名，而是一套方法语言：

- 几何节点驱动的生成结构
- 从点开始的空间生长
- 音频触发与时间线控制
- 点云 / 扫描 / 数字自然素材
- 让影像从屏幕里“长到”空间里

按这个角度看，后面很多项目虽然不一定都叫《滴流》，但已经在共享这套语言。

这里我会明确说，这是对创作脉络的总结，不是严格的版本命名。

可以举几个例子：

- `Timer` 更像是音画控制与时间结构的技术底座
- 《滴流》把这套底座推进到完整的作品哲学和空间叙事
- Kashiwa Daisuke 等现场合作中的部分舞台视觉，可以看作这套生成语言向演出现场的外溢
- 《观察与共生》、`Digital Garden`、扫描点云相关实践，则把“点、流动、空间生长”推进到记忆空间和数字自然方向
- Vision Pro 和实时引擎规划，则是在把这套语言继续推进到可进入、可游走、可多人共在的空间中

### 这一段直播里可以直接这么说

《滴流》不是做完一个版本就结束了，它更像一套母语。后面有的是正式版本，有的是方法分支。版本在作品内部升级，分支在别的项目里继续长。

## 十一、第九层：为什么最后会收束到 `SpacePort` 和 `SceneForge`

如果把前面这些都串起来，最后就会回到一个非常现实的问题：

**这些经验到底要怎样继续推进。**

我的答案现在越来越明确：

- `VIRTURA-SpacePort` 负责入口、档案、组织公开层、叙事门面
- 知识网络负责保存理论、方法和练习反思
- `SceneForge` 负责真正长期推进的工具主线

这意味着什么？

意味着前面那些太空站、虚拟航线、虚拟空间、作品试验、知识文档，不再是互相竞争的主线，而是被重新放回一个更清晰的结构里。

### 这一段可以压缩成一句话

入口归入口，档案归档案，理论归理论，工具主线只保留一个。

## 十四、结尾可以这样讲

### 标准版

如果让我总结今天这个 GitHub SpacePort 项目的脉络，我会说：前几年我一直在通过不同的项目寻找数字空间语言，那个阶段确实有很多发散的命名、实验和世界观外壳；但这些东西最后并没有白做，它们帮助我把空间感知、音画关系、数字自然、扫描记忆和工具判断慢慢积累起来。今天我更明确地知道，哪些东西应该作为档案继续保留，哪些东西应该沉淀成理论，哪些东西应该收束成真正可持续推进的工具，而《滴流》正是这条线里最清楚的母体样本。

### 更短版

我现在不是在继续扩张一个越来越大的系统，而是在把过去几年关于作品、空间、理论、团队和工具的探索，压缩成一个更清楚的结构。`SpacePort` 负责公开入口和档案，知识网络负责方法沉淀，真正的工具主线只保留 `SceneForge`；而《滴流》就是这条收束路径里最关键的作品。
