# SpacePort 与《滴流 Drop Flow》项目脉络分享讲稿

> 适用于直播分享、公开介绍、30-60 分钟项目脉络讲述

---

## 一、开场：这场分享要讲什么

大家好，今天这场分享我想讲的，不只是《滴流 Drop Flow》这一件作品本身，而是借《滴流》去讲清楚整个 VIRTURA / SpacePort 项目的脉络。

如果用一句话概括，这条线经历的是一个很典型的过程：**从早期关于虚拟空间、太空站、虚拟航线的发散探索，逐渐收束到 GitHub 上的公开档案、理论整理和工具主线，最后明确哪些内容应该留在 `VIRTURA-SpacePort`，哪些内容应该真正收束成可持续推进的工具。**

《滴流》是理解这条线最好的样本，因为它完整呈现了我们如何从技术 demo，走到创作哲学，再走到更成熟的空间叙事和工具判断。

---

## 二、第一部分：最早的起点，不是做平台，而是找空间语言

最开始吸引我们的，并不是“做一个软件产品”，而是一个更前置的问题：**能不能把数字空间当作真实空间来对待。**

所以在更早期，VIRTURA 里面自然会出现很多东西，比如：

- 虚拟航线
- 太空站
- 可体验数字空间
- 世界观包装
- 沉浸式叙事容器

这些内容现在回头看，确实显得有些发散，但它们并不是完全无效的。它们真正的价值，是帮我们确认了自己长期真正关心的核心命题：

- 空间感知
- 数字自然
- 沉浸式体验
- 人和数字空间的关系

也就是说，那个阶段虽然还没有非常清晰的产品边界，但已经开始为后面的作品和方法论提供原始语汇。

### 这一段可以压缩成一句话

我们一开始不是在做工具，而是在为未来数字空间寻找语言。

---

## 三、第二部分：GitHub 阶段，开始把发散经验收进结构里

后来进入 GitHub 阶段之后，事情开始变得更清楚一些。

我们不再只是“有很多想法”，而是开始把这些想法做成可以公开积累的结构，包括：

- 作品归档
- 项目脉络
- 知识网络
- 理论文稿
- 团队入口
- 项目导航

这一步的关键变化是，很多原本只存在于脑中的经验、现场中的判断、临时性的想法，第一次被写下来、归档下来，并且彼此建立连接。

如果说早期是在“造世界”，那么 GitHub 阶段其实是在**给这个世界建立地图、建立档案、建立认知装备。**

但问题也很明显：那时候的 `SpacePort` 一度还是承担了太多角色。

它像主港、像容器、像组织入口、像理论仓、也像产品总仓。它可以解释很多东西，但不够利于执行，也会让外部误以为这是一个很大的平台系统。

### 这一段可以压缩成一句话

GitHub 阶段最大的价值，不是把项目做大，而是第一次把很多零散经验写成结构。

## 四、第三部分：为什么理论不是悬空概念

这里我想特别强调一下理论层，因为如果只看仓库结构，很容易觉得 `our-theory` 只是一些概念性写作。

但对我们来说，这些理论真正的作用，是把原本零散的创作经验、教学经验和现场经验，收束成一套更有效的系统。

目前大概有三条最重要的理论线：

### 1. 空间原生创作框架

这条线主要回答一个问题：

**视觉创作者如何从认知走到空间作品落地。**

它强调几件事：

- 创作优先，技术为表达服务
- 先做最小闭环，再慢慢扩展
- 把复杂创作拆成可执行、可验证的小单元

这不是单纯的课程框架，而是一种从创意到作品的工程化创作方法。

### 2. 工程控制论

这条线主要回答的是：

**怎样把实践经验变成一个可反馈、可调参、可迭代的控制系统。**

以前我们对很多“感知体验设计”的理解其实更偏概念，我们知道自己想要什么氛围、什么空间感、什么人机关系，但不够知道如何稳定地复现它、拆解它、教学它。

工程控制论让我们开始用更清楚的方式理解这件事：

- 目标是什么
- 变量有哪些
- 执行怎么落地
- 反馈怎么回收

也就是说，感知体验不再只是抽象描述，而变成一个可以不断调整的闭环。

### 3. 知识网络 / 智力装备逻辑

这条线主要回答的是：

**为什么短期工作坊、项目复盘、工具经验不能只写成文章，而要变成应用态知识。**

知识网络负责存储，智力装备负责调用。

也就是说，理论在这里不是“显得宏大”，而是为了让经验能再次被激活、被复用、被组装。

### 这一段可以压缩成一句话

理论的作用，不是包装项目，而是把零散经验压缩成一套可以继续生长的方法。

## 五、第四部分：为什么 7 天工作坊不够，必须补上后半段

我们也做过关于 7 天工作坊的设计和反思，这部分其实非常重要，因为它直接暴露了一个现实问题：

**短期工作坊可以让人快速进入一个方向，但如果没有后续支撑，学员很容易学完即忘，项目经验也很难沉淀成长期能力。**

所以现在回头看，一个完整系统应该是这样的：

- 工作坊负责点火
- 知识网络负责存储
- 智力装备负责按困境调用
- 工程控制论负责反馈迭代

换句话说，7 天工作坊不是终点，它只是第一次激活。

这一点对组织工作也很重要，因为它意味着我们不能只做“看起来很厉害的短期课程”，而是要能把短期进入感，变成长期可复用的方法。

### 这一段可以压缩成一句话

工作坊解决的是“快速进入”，知识网络和反馈系统解决的是“持续留下来”。

## 六、第五部分：《滴流》为什么是这条线里最关键的样本

《滴流》之所以重要，不只是因为它拿了奖或者做了多个版本，而是因为它几乎把这条线的所有核心问题都串起来了。

### 1.0 原型版

1.0 解决的是一个很基础但很关键的问题：

**实时音画到底能不能成立。**

这一版的意义主要有两个：

- 技术 demo 成立了
- 创作哲学也第一次确定了

这个创作哲学就是：**drop 之后不是爆发，而是激烈之后的空缺、下坠之后的平静，以及从一滴水开始的持续流动。**

### 2.0 UFO 内部实验版

2.0 开始把系统往更复杂的方向推：

- AI 内容
- 现实扫描
- 多素材协同
- 现场执行

这一版的关键价值，不是“已经成熟”，而是它第一次把系统的问题暴露出来了，比如标准化不足、现场路径错误、长视频播放卡顿等等。

也正因为这些问题暴露了，后面才会开始更明确地思考：我们真正需要的是更稳定、更可复用的工具结构。

### 3.0 Rooooooom719 版

3.0 是《滴流》第一次更成熟地进入公开演出场景。

这时候《滴流》已经不只是一个实验，而是开始形成更明确的公开语言，包括：

- 数字信息流潜艇乘务组的叙事设定
- 更成熟的包裹式空间感
- 更清晰的观众感知节奏

这一版开始证明，前面的探索是可以在真实观众面前成立的。

### 杭州双年展版

杭州双年展版是《滴流》第一次把更完整的人机关系讲清楚。

这里的核心命题已经变成：

**从屏幕框定的时代，进入无边际媒体时代。**

这个版本里，球形全景视频、VR 头显、屏幕破裂、数字焦虑到数字共生的叙事，开始把作品从“流动感”推进到更完整的当代艺术语境。

### 重庆版

重庆版进一步验证了一件事：

**同一套内容结构，能不能跨空间装置迁移。**

这就让《滴流》不仅仅是一件作品，而开始变成一种内容结构和空间适配方法的验证。

### 这一段可以压缩成一句话

《滴流》不是一串独立版本，而是一条从技术 demo 到创作哲学，再到空间叙事与适配方法的收束路径。

## 七、第六部分：今天为什么要收束到 SpacePort 与 SceneForge

现在我们回头看，很多前期内容其实都可以被重新解释。

比如太空站、虚拟航线、可体验数字空间，它们并不只是无效膨胀，而是前期的：

- 感知实验场
- 叙事实验场
- 空间原型场

但今天更成熟的做法，是把这些东西拆开处理：

- `VIRTURA-SpacePort` 留作公开入口、档案数据库、叙事门面
- 理论和方法留在知识网络
- 真正长期推进的工具主线，只保留 `SceneForge`

这不是把项目做小了，而是把边界做清楚了。

我们现在更明确地知道：

- 什么应该作为前史和档案保存
- 什么应该作为理论和方法继续长
- 什么必须被做成真正能持续推进的工具

### 这一段可以压缩成一句话

入口归入口，档案归档案，理论归理论，工具主线只保留一个。

## 十、结尾版本

### 标准版

如果用一句话总结现在的 SpacePort 项目脉络，那就是：我们经历过一次非常典型的“先发散、再收束”的过程。早期我们在为未来数字空间寻找语言，后来我们通过 GitHub 把作品、理论、知识网络和组织经验逐步归档下来，而现在我们终于更明确地知道，哪些内容应该保留为入口和档案，哪些内容应该被收束成真正可持续推进的工具。

### 更短版

我们现在不是在继续扩张一个“太空港式大系统”，而是在把过去几年关于空间、感知、作品、工作坊和知识网络的探索，收束成一个更有效的组织结构：入口和档案留给 `SpacePort`，理论留在知识网络，真正的工具主线只保留 `SceneForge`。
