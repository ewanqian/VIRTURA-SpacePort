# 图片与素材库管理说明

> 适用于 `drop-flow-series` 及后续相关项目的图片、海报、截图、证书、视频封面管理

---

## 最省事的做法

不要把所有图片长期堆在一个文件夹里。

最稳的方式是：

1. 先把新收到的图片、朋友圈截图、聊天记录图、现场手机拍摄图全部丢进 `assets/_inbox/`
2. 每次整理时，再按“项目 / 版本 / 类型”分到正式文件夹
3. Markdown 文档只引用正式文件夹里的图，不引用 `_inbox`

这样做的好处是：

- 先收集，不打断整理节奏
- 正式文档不会引用到临时文件
- 后面重命名和筛选更容易

```text
    stage/
    hologram/
    social/
  timer/
    stage/

```

类型建议统一成这几类：

- `stage/`：现场图、演出图、舞台照片
- `rehearsal/`：排练图、工作照
- `social/`：朋友圈截图、公众号配图、视频号封面截图
- `docs/`：海报、票证、证书、活动页截图
- `render/`：渲染图、设计稿、效果图
- `hologram/`：纱幕、特殊装置、局部结构图

## 推荐命名规则

统一用：

```text
YYYYMMDD_项目名_地点_类型_序号.ext
```

例如：

```text
20251021_kashiwa-shenzhen_stage_wide_01.jpg
20251019_dropflow-hangzhou_stage_vr_01.jpg
20251108_dropflow-luminous_poster_01.jpg
20250514_dropflow-ufo_social_wechat_01.jpg
```

这样做有三个好处：

- 排序时天然按时间排
- 一眼能看出项目和用途
- 后面批量引用、搜索、替换都很快

---

## 每张图至少要补的最少信息

如果一时没空写完整说明，至少记住这 4 件事：

- 这是哪个项目
- 这是哪个版本或场次
- 图是现场图、海报还是截图
- 来源是自己拍的、朋友圈、官方页还是观众拍摄

最简单的做法是在每个项目文件夹里放一个 `index.md`，只写一个小表：

| 文件名 | 内容 | 来源 | 是否可公开 |
|------|------|------|-----------|
| 20251019_dropflow-hangzhou_stage_vr_01.jpg | 开幕式 VR 头显段落 | 自拍 | 可公开 |

## Markdown 里怎么嵌图

直接用相对路径：

```md
![杭州双年展开幕式现场图](./assets/drop-flow-hangzhou/stage/20251019_dropflow-hangzhou_stage_vr_01.jpg)
```

如果图片就在当前 `assets/` 根目录下，也可以直接这样写：

```md
![流光绘影海报](./assets/2025“流光绘影”光影科技艺术节海报.jpg)
```

## 图片筛选顺序建议

每个项目先找这 5 类，优先级最高：

1. 一张最能代表现场空间关系的全景图
2. 一张最能说明核心装置或视觉结构的中景图
3. 一张最适合社交媒体传播的竖图
4. 一张官方海报或活动页截图
5. 一张能证明成果的证书、新闻页或官方报道截图

如果你素材很多，不要一开始全塞进文档。先每个项目挑 3 到 5 张最强的图。

---

## 公开来源图片怎么处理最高效

如果是 B 站、公众号、视频号、官方站搜到的图，不建议直接热链外部图片。

更稳的做法是：

1. 先把来源链接记到公开资料索引里
2. 再把你确认要用的截图或保存图放进对应项目文件夹
3. 文档里优先嵌本地图片，同时在图下方附来源链接

这样后面外链失效，文档也不会空掉。
